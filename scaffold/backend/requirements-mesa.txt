# Mesa, installed deliberately WITHOUT its dependency tree.
#
#     pip install --no-deps -r requirements-mesa.txt
#
# WHY --no-deps (P0.2, measured 19 July 2026):
# mesa 2.4.0 declares matplotlib, mesa-viz-tornado and solara. The solara/Jupyter tree it pulls
# writes files such as
#   share/jupyter/labextensions/@jupyter-widgets/jupyterlab-manager/static/vendors-node_modules_...js.map
# whose paths exceed the Windows 260-character limit. On a machine without LongPathsEnabled the
# install aborts with:
#   ERROR: Could not install packages due to an OSError: [Errno 2] No such file or directory
# Because pip resolves and downloads before installing, this failed the ENTIRE documented install,
# not just the visualisation extras.
#
# MERIDIAN uses none of that tree. mesa is a vestigial base class here: no scheduler, no AgentSet,
# no visualisation — `engine.py` iterates plain lists. matplotlib, mesa-viz-tornado and solara are
# never imported by this codebase.
#
# mesa's genuine import-time requirements (tqdm, numpy, pandas) are listed in requirements.txt so
# they are installed by the first command.
#
# VERIFIED 19 July 2026, CPython 3.12.10 on Windows 10 with LongPathsEnabled=0:
#   `import mesa` -> mesa 2.4.0 OK, and `python -m pytest tests -v` -> 5 passed in 0.93s.
# `pip check` will report the three visualisation packages as missing. That is expected and
# intended; it is not a broken install.
#
# NOTE: whether mesa remains the ABM substrate at all is an open owner decision (current-state
# audit section 8, item 3). If mesa is dropped, this file and the three deps above go with it.

mesa>=2.1,<3.0
