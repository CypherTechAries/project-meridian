[`../../docs/PLAN.pdf`](../../docs/PLAN.pdf) ┬з6 for the stack rationale and ┬з2 for the
open-source licensing audit). Each decision is recorded here so future contributors тАФ human or
AI тАФ inherit the *why*, not just the *what*.

## ADR-001 тАФ Backend: Python 3.11+ / FastAPI / Pydantic v2
**Status:** Accepted.
The simulation core, ABM libraries, and LLM tooling are all strongest in Python. FastAPI gives
async REST + native WebSocket support and auto-generated OpenAPI docs with minimal ceremony.
Pydantic v2 is the schema source of truth тАФ its models generate the JSON Schema mirrors in
`/schemas` and validate every object crossing the API and LLM boundaries.

**Amendment тАФ 19 July 2026 (P0.1).** The stack choice stands. The final sentence describes the
intended schema pipeline in the present indicative and does not hold today, in two respects.

1. **"its models generate the JSON Schema mirrors in `/schemas`."** The nine files in
   `../schemas/` are not generated by anything in the tree: no call to `model_json_schema()` or
   any other schema-export API exists under `../backend/app/`, and no generation script or build
   step is present. The mirrors are hand-maintained and can drift from the Pydantic models
   without detection тАФ which is why `CLAUDE.md` instructs that all three be kept in sync by hand.
2. **"validate every object crossing the API and LLM boundaries."** Pydantic does validate
   request and response models on the REST routes. It does **not** validate the published JSON
   Schema mirrors at runtime: no JSON Schema validation library is present in the scaffold
   (`grep -rn "jsonschema" scaffold/` returns nothing), so those nine files validate nothing.
   And nothing currently crosses an LLM boundary: no module under `../backend/app/` imports a
   model client or any HTTP client, so the LLM half of the sentence is untested rather than
   enforced. Read the sentence as the intended contract.

## ADR-002 тАФ Simulation engine: Mesa (Apache-2.0)
**Status:** Accepted.
Mesa is a mature, permissively licensed (Apache-2.0) agent-based-modelling framework. It gives
us a `Model`/`Agent` structure and scheduling without inventing our own. We use it for the
meso (cohort) and micro (institutional) tiers; the macro tier is a plain seeded rules object,
not an agent, because national aggregates are not usefully modelled as a stepped agent.
Rationale for permissive licence: the product may be commercialised тАФ GPL engines were ruled
out by the licensing audit at `../../docs/PLAN.pdf` ┬з2.

**Amendment тАФ 19 July 2026 (P0.1).** The permissive-licensing rationale holds for the engine
choice, but the stack is not uniformly permissive: `psycopg2-binary`
(`../backend/requirements.txt:17`) is **LGPL** and is absent from the `PLAN.pdf` ┬з2 audit. The
closed audit records it as commercially benign under the SaaS model `PLAN.pdf` assumes
(dynamic linking, no distribution). Recorded so the inventory is not read as complete.
Separately: whether Mesa remains the ABM substrate is an open owner decision (closed audit ┬з8
item 3) and is not settled by this amendment.

## ADR-003 тАФ Database: PostgreSQL via SQLAlchemy 2.0
**Status:** Accepted.
We persist three things: `simulation_run` (seed + scenario), immutable `state_snapshot` per
tick, and an append-only `event_log`. JSON columns store schema objects verbatim so the DB is
generic across nation archetypes (data, not code тАФ see ADR-007). Snapshots make scenario
branching and Monte-Carlo batch runs cheap: fork from any tick. SQLAlchemy chosen over raw SQL
for typed models that mirror the Pydantic schemas.

**Amendment тАФ 19 July 2026 (P0.1).** "We persist three things" was recorded in the present
indicative and is **not implemented in the scaffold**; this is publication blocker B3. **Nothing
is written to a database.** `SimulationRun`, `StateSnapshot` and `EventLog` are defined at
`../backend/app/db/models.py:23`, `:42` and `:56` and are **instantiated nowhere** тАФ no
`session.add`, `session.commit` or `session.merge` call exists anywhere in the scaffold, and
`get_db()` (`../backend/app/db/session.py:44`) is used by no route. The only database operation
attempted is table creation at start-up, inside an `except Exception` that logs "Running
in-memory only." and continues (`../backend/app/main.py:30-39`), so the application starts
identically with or without a database. All run state lives in the in-process `_RUNS` dictionary
(`../backend/app/api/runs.py:18`) and does not survive a restart. The decision stands as the
intended persistence design; the schema is designed and unwired. Read "we persist" as "we intend
to persist", matching the "in the scaffold" convention already used in ADR-004 and ADR-005.

## ADR-004 тАФ LLM gateway: LiteLLM-ready, stubbed by default (MIT)
**Status:** Accepted.
`llm_gateway.py` is an interface, not an implementation, in the scaffold: it returns canned
text so the system runs with **no API keys**. LiteLLM (MIT) is the intended router because it
exposes one interface over local *and* frontier models тАФ micro-agent reasoning can go to a
cheap/local model while briefings go to a frontier model, tuned per call. Swapping the stub for
real calls is a localized change to one module.

## ADR-005 тАФ Real-time: FastAPI WebSocket now, Socket.IO-ready
**Status:** Accepted.
Tick broadcast uses a plain FastAPI WebSocket endpoint for the scaffold (zero extra infra).
`python-socketio` is already a dependency so a full build can move to rooms/namespaces for
multi-player shared runs without changing the engine. The engine is transport-agnostic: it
exposes `step()` and `macro_snapshot()`; the transport layer decides how to broadcast.

## ADR-006 тАФ The determinism boundary (the load-bearing decision)
**Status:** Accepted. **Do not weaken.**
**The LLM never mutates macro or meso numeric state directly.** It only returns structured
*proposals* (`ActionProposal`) and interpretive text (briefings). `engine.py` is the sole
writer of numeric state: it validates a proposal's legality/feasibility and computes magnitude
itself (`_validate_and_price`), deliberately treating the proposal's numeric parameters as
advisory. This is enforced *structurally* тАФ `llm_gateway.py` imports no state object, and
`test_engine.py::test_llm_gateway_cannot_write_state` guards it. Rationale: reproducibility and
explainability are core product promises; letting an LLM write numbers would destroy both.

**Amendment тАФ 19 July 2026 (P0.1).** The decision stands and must not be weakened. Three
factual statements in it do not hold today.

1. **"`engine.py` is the sole writer of numeric state."** Correct as a statement of *ownership*,
   false as a literal file claim. The engine is the only component that *decides* numeric state,
   but the assignments themselves execute elsewhere: macro values at
   `../backend/app/simulation/agents/macro_state.py:47`, cohort beliefs at
   `../backend/app/simulation/agents/cohort_agent.py:38`. Reword when citing; do not refactor.
2. **"it validates a proposal's legality/feasibility and computes magnitude itself
   (`_validate_and_price`)."** It does none of those things. Publication blocker B1. The entire
   gate is `base = ACTION_EFFECTS.get(proposal.action_type, {})` then `return dict(base)`
   (`../backend/app/simulation/engine.py:121-130`) тАФ a lookup in a fixed seven-entry table
   (`:35-43`) returning a constant. Its signature receives only the proposal, never the agent
   spec, so an agent's declared `constraints` are structurally unreachable; proven by
   substitution, a 40-tick state hash identical with and without them
   (`../../docs/delivery/A3-VERIFICATION-RESULTS.md:92-98`). The clause "deliberately treating
   the proposal's numeric parameters as advisory" **is** true and stands.
3. **"This is enforced *structurally* тАж and `test_llm_gateway_cannot_write_state` guards it."**
   The boundary is currently satisfied **trivially, because no language model is invoked at
   all** тАФ no module under `../backend/app/` imports a model client or any HTTP client. There is
   no structural guard preventing a model call path from being added: `config.py:23-25` already
   declares an `llm_mode` of "stub" or "live". And the test does not guard it: it calls the stub
   once and asserts the returned object is an `ActionProposal` with no attribute named
   `apply_deltas` and no attribute named `macro_state`
   (`../backend/tests/test_engine.py:61-77`). No import-graph analysis, no attempted mutation,
   no real model. A gateway that mutated authoritative state by any other route would pass it
   unchanged. **TARGET, not delivered:** a boundary guard that fails when the gateway package
   gains any import of or reference to an authoritative-state object, running on every change.

## ADR-007 тАФ Reproducibility: one seed, threaded everywhere
**Status:** Accepted.
Every run has a `seed`, threaded into `MeridianModel.__init__` тЖТ `self.rng`
(`random.Random`). All engine, agent, and diffusion randomness draws from that one RNG тАФ never
the global `random` or unseeded `numpy`. Same seed + scenario + decisions тЗТ identical
macro/meso numbers, proven by `test_same_seed_is_deterministic`. LLM text is explicitly *not*
part of reproducible state; it is versioned separately by `model_id + prompt_version +
temperature` and treated as an interpretive layer.

**Amendment тАФ 19 July 2026 (P0.1).** The seed-threading half is accurate: there is exactly one
`random.Random` in application code (`../backend/app/simulation/engine.py:83`) and exactly three
draw sites (`agents/cohort_agent.py:36`, `diffusion.py:75`, `engine.py:135`). Two other
statements do not hold.

1. **"Same seed + scenario + decisions тЗТ identical macro/meso numbers, proven by
   `test_same_seed_is_deterministic`."** Not proven, and the "decisions" term is misleading:
   submitted decisions never reach the tick loop (`../backend/app/api/routes_simulation.py:80-101`;
   `engine.py:147-180`). The test compares only the final macro dictionary
   (`../backend/tests/test_engine.py:34-40`) тАФ no cohort belief, no narrative adoption, no event
   log, no snapshot history тАФ and both runs call the same in-process stub, so agent output is
   held constant rather than varied. The founder-settled sentence, to be used verbatim: "The
   existing stubbed execution path reproduces the same tested numeric outputs when the seed,
   scenario and stubbed agent outputs remain identical."
2. **The versioning sentence is a target, not behaviour.** `PROMPT_VERSION = "v1"` is declared
   once (`../backend/app/simulation/llm_gateway.py:38`) and **read by nothing**. Neither
   `model_id` nor temperature is logged anywhere. Read the sentence as the intended contract for
   when a live model is wired.

**Superseded in part.** This ADR does not address randomness *isolation*, and the single shared
stream it describes is itself a defect: there are no named substreams, so adding a draw in one
subsystem silently shifts every later draw everywhere else. That is why perturbing cohort counts
appears to move macro numbers when no causal channel exists
(`../../docs/delivery/A3-VERIFICATION-RESULTS.md:142-175`). The replacement architecture is Phase 0
item **P0.4A**, created by founder decision of 18 July 2026 and ordered P0.4 тЖТ P0.4A тЖТ P0.5. The
replacement architecture is recorded in
[`ADR-010 тАФ Deterministic randomness architecture`](../../docs/adr/ADR-010-deterministic-randomness-architecture.md),
accepted by the founder on 19 July 2026. **Nothing in ADR-010 is implemented**, so ADR-007
continues to describe the code as it stands today; ADR-010 governs what replaces it.

## ADR-008 тАФ Nation types are data, not code
**Status:** Accepted.
Per `../../docs/PLAN.pdf` ┬з9 (the nation-archetype expansion framework), a nation archetype is a
scenario-template JSON (starting
macro values, institutional roster, relationship-graph seed, crisis hooks, win/loss set). The
engine contains **no** archetype-specific branches. If the engine appears to need a special
case, that is a signal the schema is incomplete тАФ extend the schema, not the engine. Adding an
eighth archetype must require only a new `scenarios/*.json`.

**Amendment тАФ 19 July 2026 (P0.1).** The decision stands as the intended extensibility model. As a
statement of current fact it is **unverified in one half and false in the other**; this is
publication blocker B4.

1. **"Adding an eighth archetype must require only a new `scenarios/*.json`" is untested.**
   `../scenarios/` contains exactly one file, `kestral-strait.json`. No second scenario has ever
   been constructed or run in any recorded pass, so whether an arbitrary conforming file loads and
   completes a run is not known. Scenario loading is a path-existence check followed by
   `json.loads` with no content validation (`../backend/app/api/runs.py:21-26`); cohort records are
   then parsed through Pydantic (`../backend/app/simulation/engine.py:98-101`).
2. **"The engine contains no archetype-specific branches" is accurate as far as it was checked,
   and that is not far enough to support the promise.** `scenario_id` appears in
   `../backend/app/simulation/engine.py` only at `:52` and `:86`, as a value read and stored, never
   as a condition. That check covered `engine.py` only; `agents/`, `diffusion.py` and the schema
   layer were not searched for archetype conditionals.
3. **What is directly verified is worse than a missing branch: most archetype content is read by
   no code at all.** A twelve-term search over `../backend/app/` for `media_exposure`,
   `demographics`, `religion_majority`, `primary_language`, `existing_grievance`, `target_cohorts`,
   `amplification_network`, `detection_probability`, `attribution_probability`,
   `desired_behaviour`, `perceived_independence` and `truth_status`, excluding the schema
   definition file, returns zero hits for all twelve. `represents_population` occurs only at its
   declaration (`../backend/app/simulation/schemas/agent_schema.py:95`) and in scenario data.
   Two archetypes differing only in those fields will behave identically.
4. **The failure is silent.** An action type absent from `ACTION_EFFECTS`
   (`../backend/app/simulation/engine.py:35-43`) resolves to an empty delta, and `apply_deltas`
   (`../backend/app/simulation/agents/macro_state.py:36-41`) skips any key it does not recognise
   and any value that is not a top-level number тАФ no error, no warning, no return value. New
   scenario data can therefore be loaded and have no effect, undetectably.

**TARGET, not delivered:** archetype content that causally drives behaviour, with a strict mode
that raises on unknown or unreachable effect keys instead of skipping them.

## ADR-009 тАФ Diffusion model for narrative spread
**Status:** Accepted.
Information-campaign effectiveness is computed by a seeded Linear Threshold diffusion over a
`networkx` cohort graph (`diffusion.py`), against each cohort's `influence_susceptibility` and
`network_position` тАФ **not** authored by the LLM. The LLM may compose campaign *content*
(channel, messenger framing) constrained to the `Campaign` schema; the engine decides whether
and how far it spreads. Consistent with ADR-006.

**Amendment тАФ 19 July 2026 (P0.1).** The decision stands. Three statements in it describe intended
behaviour rather than current behaviour.

1. **The diffusion step runs, but its output does not reach cohort beliefs.** `_step_diffusion`
   assigns the result of `linear_threshold_step` to `self.narrative_adoption`, a plain dictionary
   on the model (`../backend/app/simulation/engine.py:138-145`). The only cohort field written
   anywhere in the scaffold is `beliefs.government_competence`, decremented by grievance drift
   (`../backend/app/simulation/agents/cohort_agent.py:35-38`). The scaffold's own
   `AGENT_TASK_TEMPLATE.md:48-49` already records this. `Narrative` is never instantiated, so
   `Narrative.adoption_by_cohort` is never written despite the `diffusion.py:5` docstring.
2. **"The LLM may compose campaign *content*" is a target тАФ no such function exists.**
   `../backend/app/simulation/llm_gateway.py` defines exactly three functions: `_stub_action_type`
   (`:41`), `propose_action` (`:54`) and `generate_briefing` (`:85`). There is no
   campaign-composition function, `Campaign` is never instantiated, and every field of
   `CampaignMetrics` sits at its 0.0 default.
3. **"Adoption is monotonic non-decreasing" (`../backend/app/simulation/diffusion.py:51`) does not
   hold.** The jitter term at `:75` spans тИТ0.01 to +0.01 and is added to `gain` at `:76`, so
   adoption can fall. The clamp half of that sentence is accurate.

Recorded as fact, not as an instruction to refactor: what the ADR decides тАФ that spread is computed
by the engine and not authored by a model тАФ is unchanged and must not be weakened.
